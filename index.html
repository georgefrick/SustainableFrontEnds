Why?

  1. Avoid Churn.
  2. Beat The Learning Curve.
  3. Reliable, Scalable, and Maintainable.
  4. Credibility.

  • New app - large portion of time on foundations: security, layout, build process
  • developers are learning a new framework, not established best practices

This comes down to being able to prioritize business needs so that we can react to a changing business world. You cannot build credibility with the business if requests often mean starting over, or you give the impression you didn't plan for change. We don't always want to be rebuilding software, we want to be build a platform of incremental capability.

  • We tend to pick a framework/library, and we're done.
  • This is generally based on popularity
  • BUT, frameworks are contributors of architectural ideas
  • We have requirements, and we need an architecture based on our goals.
  • Understand that modern frameworks will in some way prescribe how we work.

Two Parts

  1. Planning our build through good architecture.
  2. Following through with best practices.
  • make more complete decisions at the beginning of a project
  • we have to consider our front end architecture the same way we do for our services
  • We then have to follow that up, with good habits while developing.
  • the goal is for the framework to be a layer in your architecture, and not the container for it.
We'll talk about the first from the architectural level, and then the second will be a series of tips we'll look at.

Security

  • Users
  • Data
  • Company wide concern
  • Principle of least privilege
  • data should never arrive from server
  • server should block actions
  • secure your application and be part of any security planning and discussions
I want to mention security briefly; it shouldn't be as much the concern of the front end architect as the back end. It should be a company wide concern, and be based on the principle of least privilege. You should not hide data the user doesn't have access to; it should never arrive from the server. You should not hide actions the user can't take - someone will fire up postman and send the request by hand. Security has to be an onion.

    2. Approach

    Let's finally talk frameworks. Maybe your project is small and you don't have any special requirements. Maybe you think this step is "done", because you are "A React house" and you build everything in React.
    But let's just take a look at some architectural alternatives you might make based on your information. For example, a lot of bloggers and small companies look at their requirements and choose Word Press, and they experience a lot of success.

    Static Website

    • Fast
    • Cheap
    • SEO
    • Deployment
    		    
    • Complexity
    • Change
    • Dynamic Content
    • Temporary websites, mock-ups, internal information pages.
    Fast, affordably developed site with easy SEO and deployment. This is ideal for temporary websites, or information pages that require their own design. While many people will use mocking tools; I prefer to build out a static website - users can see navigation and get a feel for how they’ll use the app.

    Static - Server Generated

    • Benefits of Static
    • Databases, Webservices, etc
    • Performance
    • Complexity
    • Change
    • Interactivity
    • Status screens, reporting, internal information sites, blogs, news feeds, and kick-start.
    You can build generated static sites using frameworks like Jekyll or wintersmith and languages like Go. Sometimes, users desire more interactivity. Handle change better than a static site You can get information from a database, web service, without the security concerns of api traffic. Another benefit of using a static site generator, as the templates and any code can be used to kick-start a more in-depth project. This makes for an inherently more secure and performant dashboard - just regenerate the site on a cycle and don't worry about what the front end is allowed to do.

    Content Management System

    • Out of box features
    • Low cost
    • Fast start
    • User managed content
    • Maintenance
    • Customization
    • User Learning Curve
    • Long Term Business Problems
    • Creative content and publicly facing pages with changing designs.
    It’s strange to be suggesting this. You need non-technical people to manage the content of the site. Out of box features (polls, surveys, analytics, reports, user management). Don’t use a CMS just to start; there is Yeoman for that.

    Single Page Applications

    • "Applications"
    • Frameworks
    • Tooling
    • Customization
    • Maintenance
    • SEO
    • Memory
    • Performance
    • Highly customized business functionality, data manipulation and workspaces.
    In the majority of situations where you are planning an enterprise application; this will fit for you. Bootstrap application development, ex: Yeoman and many CLIs. Developers can quickly build out applications that support specific business functions and fit directly to requirements. More and more, the benefits of static site generation are showing up here as well. You will have trouble here with switching frameworks, aging code, security, complexity

    Multi Page Applications

    • Frameworks
    • Tooling
    • Customization
    • Complexity
    • Management
    • Highly focused business functionality, data manipulation and workspaces.
    • Not MVC
    A multipage application can still be appropriate and actually strong when you are attempting to get the advantages of SPAs and deal with specific disadvantages. There is actually a small but growing idea here that follow that of micro-services. Micro-front ends. By going back to multi page applications, but using SPA tools and frameworks; we can minimize problems like architectural change, SEO, and complexity. Traditionally a multi-page application would imply MVC and would be tied to some kind of non-restful (or non graph-ql) back end. Forcing developers to work on both, and I don’t recommend that approach. What I can recommend is there can be benefits to building more, smaller applications. As we’ve seen through this talk - you can identify and architect each application according to it’s needs. MPAs have the disadvantage of complexity of management. As the number of projects goes up, maintenance, new features and overall organization can suffer. It takes an approach that manages the applications as a portfolio to do this well.

    Don't Inline Logic

    • Locks you into framework
    • Duplication!
    • Testing
    • Modularity
    

@Injectable()
export class RoleGuard implements CanActivate {

  constructor(private router: Router, private authService: AuthService) {}

  public canActivate(routeSnapshot: ActivatedRouteSnapshot,
                      state: RouterStateSnapshot): Observable<boolean> | boolean {

    const neededRole = routeSnapshot.data[“role”];

    if( neededRole === undefined ) {
       return true;
    }

    const roles = this.authService.getCurrentUserRoles();
    const role = roles.find( eachRole => eachRole.roleName === neededRole );
    return (role !== undefined);
  }
}
					
    

@Injectable()

export class RoleGuard implements CanActivate {

  constructor(private router: Router, private authService: AuthService) {}

  public canActivate(routeSnapshot: ActivatedRouteSnapshot,
                             state: RouterStateSnapshot): Observable<boolean> | boolean {

    const neededRole = routeSnapshot.data[“role”];

    return this.authService.isAuthorizedForRole(neededRole); // Observable!
  }
}

    Manage Dependencies

    • Monday morning headaches
    • Broken CI Pipelines
    • Dependency Hell

    For each library, helper, tool, or API you bring into your application, you are adding a dependency.

    Don't use it.

    • Is it actively maintained?
    • How is it tested?
    • Does it solve a complete set of problems?
    • It can be found as part of a larger, more useful library.
    • It can be found as part of a smaller, more specific library.

    For each library, helper, tool, or API you bring into your application, you are adding a dependency.

    Build Libraries

    • Code Reuse & Sharing
    • Independence
    • Framework Freedom
    • Proper Retirement
    • Future Planning

    • Code Reuse - across projects and frameworks and the frontend/backend.
    • Independence - libraries can be independently worked on and tested.
    • Framework Freedom - code can be built to work with a framework (calendar), or alone (calculations)
    • Proper Retirement - code can be retired as appropriate; such as replacement with a more powerful commercial library.
    • Future Planning - if you are starting something new, you can plan which libraries to use or replace.
    • Web Components - Polymer, StencilJS, etc.
